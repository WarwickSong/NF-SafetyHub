# SafetyHub 交付运行手册总览

## 目标范围

本目录只面向两类运行方式：

- 当前测试部署：在当前服务器使用 `.venv` 和 `run_prod.sh` 启动 SafetyHub。
- 内网服务器 Docker Compose 部署：在 VPN/堡垒机访问的内网服务器上部署 SafetyHub、PostgreSQL 和 Nginx。

其他历史部署路线已从本交付目录移除，避免误用。

## 文件清单

| 文件 | 用途 |
|------|------|
| `.env.current.example` | 当前测试环境模板 |
| `.env.production.example` | 生产/内网 Docker Compose 环境模板 |
| `setup_venv.sh` | 当前测试服务器创建 `.venv` 并安装依赖 |
| `run_dev.sh` | 当前测试服务器调试启动入口 |
| `run_prod.sh` | 当前测试服务器生产形态启动入口，默认监听 4000 |
| `export_apikeys_for_intranet.sh` | 从当前 PostgreSQL 导出 `api_keys` 表数据 |
| `deploy_intranet_docker.sh` | 内网服务器 Docker Compose 部署并导入 APIKeys |
| `内网服务器部署说明.md` | 内网 Docker Compose 部署完整步骤 |

## 当前测试部署

在当前服务器执行：

```bash
cd /var/www/NF-SafetyHub
cp 交付运行手册/.env.current.example .env
```

补齐 `.env` 中的占位符：

```env
KEY_PROVIDER_PASSWORD=replace-with-current-key-provider-password
KEY_PROVIDER_AUTH_VERSION=replace-with-current-key-provider-auth-version
```

启动：

```bash
./交付运行手册/run_prod.sh
```

默认访问：

```text
http://当前服务器IP:4000/admin/
```

## 内网服务器 Docker Compose 部署

按 [内网服务器部署说明.md](./内网服务器部署说明.md) 执行。

最简流程是在当前服务器先确认离线包配置文件，然后生成完整离线包：

```bash
cd /var/www/NF-SafetyHub
vim 内网离线部署包/intranet.env
bash 交付运行手册/build_intranet_offline_bundle.sh
```

把生成的 `safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz` 上传到内网服务器后：

```bash
cd /opt
tar -xzf safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz
cd safetyhub_intranet_bundle_YYYYMMDD_HHMMSS
bash install.sh
```

离线包已经包含 SafetyHub、PostgreSQL 和 Nginx 镜像，以及 APIKeys SQL、Compose 配置和自动部署脚本。

默认对外端口为 443。服务器本机验证：

```text
http://127.0.0.1/admin/
```

如果内网网关或负载均衡负责 HTTPS 证书终止，用户侧访问地址通常是：

```text
https://内网域名/admin/
```

如果需要使用其他对外端口，修改 `.env`：

```env
SAFETYHUB_HTTP_PORT=其他端口
```

## 数据盘配置

内网服务器 `/home` 挂载在独立数据盘，默认数据盘路径：

```env
SAFETYHUB_DATA_ROOT=/home/safetyhub
SAFETYHUB_DATA_DIR=/home/safetyhub/app
SAFETYHUB_POSTGRES_DATA_DIR=/home/safetyhub/postgres
```

如果实际挂载点后续变化，只修改这三个值即可。

## 数据迁移边界

迁移时只迁移 `api_keys` 表。

目标内网库会清空以下运行数据表：

```text
admin_operations
approval_requests
approval_chains
audit_logs
image_assets
message_archives
security_policies
```

目标端 `api_keys` 会在同一个 PostgreSQL 事务中清空并重新导入，导入失败会回滚。
