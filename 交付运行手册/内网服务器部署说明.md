# SafetyHub 内网服务器离线部署说明

## 最简结论

可以在当前服务器生成一个完整离线部署包，然后把这一个 `.tar.gz` 上传到内网服务器。

内网服务器只需要：

```bash
cd /opt
sha256sum -c safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz.sha256
tar -xzf safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz
cd safetyhub_intranet_bundle_YYYYMMDD_HHMMSS
bash install.sh
```

离线包会把 `/var/www/NF-SafetyHub/内网离线部署包/intranet.env` 复制为包内 `NF-SafetyHub/.env`，账号、密码、上游中转站和 Key Provider 配置都会被带过去，不需要在内网服务器现场填写。

## 离线包包含什么

离线包由 `build_intranet_offline_bundle.sh` 生成，包含：

```text
safetyhub-images.tar
NF-SafetyHub/
NF-SafetyHub/.env
install.sh
```

默认升级包不包含、不导入 `data/safetyhub_apikeys_YYYYMMDD_HHMMSS.sql`。本次内网部署会继续使用内网 PostgreSQL 中已有的 `api_keys` 表，部署脚本会删除并重建当前系统需要的其他业务表。

`safetyhub-images.tar` 包含：

```text
nf-safetyhub-safetyhub:latest
postgres:16-alpine
nginx:1.27-alpine
```

所以内网服务器不需要访问 Docker Hub。

## 一、源服务器生成离线包

在当前服务器执行：

```bash
cd /var/www/NF-SafetyHub
bash 交付运行手册/build_intranet_offline_bundle.sh
```

生成结果默认位于：

```text
/var/www/NF-SafetyHub/内网离线部署包/safetyhub_intranet_bundle_YYYYMMDD_HHMMSS/
/var/www/NF-SafetyHub/内网离线部署包/safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz
/var/www/NF-SafetyHub/内网离线部署包/safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz.sha256
```

生成脚本会：

- 构建 SafetyHub 镜像；
- 打包 SafetyHub、PostgreSQL 和 Nginx 镜像；
- 复制完整运行代码和 Compose/Nginx 配置；
- 复制 `/var/www/NF-SafetyHub/内网离线部署包/intranet.env` 到包内 `NF-SafetyHub/.env`；
- 生成内网一键安装脚本 `install.sh`。

本次内网升级不导出、不导入 API Key。内网服务器上已有的 `api_keys` 表由部署脚本保留继续使用。

如果构建机已经有 `postgres:16-alpine` 和 `nginx:1.27-alpine`，且不想联网拉取，可执行：

```bash
SAFETYHUB_SKIP_PULL=true bash 交付运行手册/build_intranet_offline_bundle.sh
```

## 二、上传到内网服务器

通过堡垒机或文件传输工具上传：

```text
safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz
safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz.sha256
```

推荐放在内网服务器：

```text
/opt
```

## 三、内网服务器安装

在内网服务器执行：

```bash
cd /opt
sha256sum -c safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz.sha256
tar -xzf safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz
cd safetyhub_intranet_bundle_YYYYMMDD_HHMMSS
bash install.sh
```

如需在内网服务器上临时调整数据盘或对外端口，可在执行 `bash install.sh` 前修改：

```bash
nano NF-SafetyHub/.env
```

内网服务器 `/home` 挂载在独立数据盘，默认数据盘配置已设为：

```env
SAFETYHUB_DATA_ROOT=/home/safetyhub
SAFETYHUB_DATA_DIR=/home/safetyhub/app
SAFETYHUB_POSTGRES_DATA_DIR=/home/safetyhub/postgres
```

安装脚本会自动创建 `app` 和 `postgres` 目录，PostgreSQL 数据会持久化在 `/home/safetyhub/postgres`。

默认对外端口配置：

```env
SAFETYHUB_HTTP_PORT=443
```

这里的 `443` 是宿主机对外端口，容器内 Nginx 仍监听 80，Compose 会做 `443:80` 端口映射。

如果内网统一入口已经有 HTTPS 网关、负载均衡或反向代理负责证书终止，SafetyHub 这边不需要额外放证书，只保持 `SAFETYHUB_HTTP_PORT=443` 即可。

如果要求 SafetyHub 容器本身直接提供 HTTPS 证书终止，则还需要额外调整 `nginx/nginx.conf`、挂载证书目录，并把 Compose 改成容器内监听 443；当前离线包默认不是这种模式。

离线镜像模式默认已经启用：

```env
SAFETYHUB_IMAGE=nf-safetyhub-safetyhub:latest
SAFETYHUB_SKIP_BUILD=true
```

## 四、安装脚本会做什么

`install.sh` 会自动执行：

- `docker load -i safetyhub-images.tar`；
- 检查包内 `NF-SafetyHub/.env` 是否存在；
- 调用 `NF-SafetyHub/交付运行手册/deploy_intranet_docker.sh`；
- 启动 PostgreSQL；
- 保留已有 `api_keys` 表，删除并重建当前系统需要的其他业务表；
- 启动 SafetyHub 和 Nginx。

本次内网升级不使用 `SAFETYHUB_IMPORT_APIKEYS`，也不从 JSON/SQL 重新导入 API Key。执行前请确认内网服务器现有 PostgreSQL 数据目录配置正确，尤其是：

```env
SAFETYHUB_POSTGRES_DATA_DIR=/home/safetyhub/postgres
POSTGRES_DB=safetyhub
POSTGRES_USER=safetyhub
POSTGRES_PASSWORD=现有内网数据库密码
DB_URL=postgresql+asyncpg://safetyhub:现有内网数据库密码@postgres:5432/safetyhub
```

如果这些值指向了新的空目录或不同数据库，部署脚本会看不到原来的 `api_keys` 表。

## 五、验证

默认对外端口是 443：

```bash
curl -i http://127.0.0.1/health/ready
curl -I http://127.0.0.1/admin/
```

浏览器通过 VPN 访问：

```text
http://内网服务器IP/admin/
```

如果内网网关或负载均衡已经配置 HTTPS 证书，用户侧访问地址通常是：

```text
https://内网域名/admin/
```

如果 `.env` 中改成其他 `SAFETYHUB_HTTP_PORT`，验证端口也要同步改。

## 六、常用运维命令

进入应用目录：

```bash
cd /opt/safetyhub_intranet_bundle_YYYYMMDD_HHMMSS/NF-SafetyHub
```

查看状态：

```bash
docker compose ps
```

查看日志：

```bash
docker compose logs -f safetyhub
```

重启：

```bash
docker compose restart safetyhub nginx
```

停止：

```bash
docker compose down
```
