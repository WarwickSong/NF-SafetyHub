# SafetyHub 内网服务器离线部署说明

## 最简结论

可以在当前服务器生成一个完整离线部署包，然后把这一个 `.tar.gz` 上传到内网服务器。

内网服务器只需要：

```bash
cd /opt
sha256sum -c safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz.sha256
tar -xzf safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz
cd safetyhub_intranet_bundle_YYYYMMDD_HHMMSS
bash install.sh
```

离线包会把 `/var/www/NF-SafetyHub/内网离线部署包/intranet.env` 复制为包内 `NF-SafetyHub/.env`，账号、密码、上游中转站和 Key Provider 配置都会被带过去，不需要在内网服务器现场填写。

## 离线包包含什么

离线包由 `build_intranet_offline_bundle.sh` 生成，包含：

```text
safetyhub-images.tar
NF-SafetyHub/
NF-SafetyHub/.env
install.sh
```

默认升级包不包含、不导入 `data/safetyhub_apikeys_YYYYMMDD_HHMMSS.sql`。本次内网部署会继续使用内网 PostgreSQL 中已有的 `api_keys` 表，部署脚本会删除并重建当前系统需要的其他业务表。新版本不再运行时写入 `message_archives`，通过请求样本写入 `training_conversations`，拦截和脱敏证据写入 `audit_logs`，图片资产索引写入 `image_assets`，报表文件写入 `${SAFETYHUB_DATA_DIR}/reports`。

`safetyhub-images.tar` 包含：

```text
nf-safetyhub-safetyhub:latest
postgres:16-alpine
nginx:1.27-alpine
```

所以内网服务器不需要访问 Docker Hub。

## 一、源服务器生成离线包

在当前服务器执行：

```bash
cd /var/www/NF-SafetyHub
bash 交付运行手册/build_intranet_offline_bundle.sh
```

生成结果默认位于：

```text
/var/www/NF-SafetyHub/内网离线部署包/safetyhub_intranet_bundle_YYYYMMDD_HHMMSS/
/var/www/NF-SafetyHub/内网离线部署包/safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz
/var/www/NF-SafetyHub/内网离线部署包/safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz.sha256
```

生成脚本会：

- 构建 SafetyHub 镜像；
- 打包 SafetyHub、PostgreSQL 和 Nginx 镜像；
- 复制完整运行代码和 Compose/Nginx 配置；
- 复制 `/var/www/NF-SafetyHub/内网离线部署包/intranet.env` 到包内 `NF-SafetyHub/.env`；
- 生成内网一键安装脚本 `install.sh`。

本次内网升级不导出、不导入 API Key。内网服务器上已有的 `api_keys` 表由部署脚本保留继续使用。

如果构建机已经有 `postgres:16-alpine` 和 `nginx:1.27-alpine`，且不想联网拉取，可执行：

```bash
SAFETYHUB_SKIP_PULL=true bash 交付运行手册/build_intranet_offline_bundle.sh
```

## 二、上传到内网服务器

通过堡垒机或文件传输工具上传：

```text
safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz
safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz.sha256
```

推荐放在内网服务器：

```text
/opt
```

## 三、内网服务器全新安装

> 本节面向**第一次部署**（内网服务器上还没有在运行的 SafetyHub 实例）。
> 如果服务器上已经有一个正在运行的 SafetyHub `docker compose`，要低中断替换升级，请改走 **第五节**，不要直接 `bash install.sh`。

在内网服务器执行：

```bash
cd /opt
sha256sum -c safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz.sha256
tar -xzf safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz
cd safetyhub_intranet_bundle_YYYYMMDD_HHMMSS
bash install.sh
```

如需在内网服务器上临时调整数据盘或对外端口，可在执行 `bash install.sh` 前修改：

```bash
nano NF-SafetyHub/.env
```

内网服务器 `/home` 挂载在独立数据盘，默认数据盘配置已设为：

```env
SAFETYHUB_DATA_ROOT=/home/safetyhub
SAFETYHUB_DATA_DIR=/home/safetyhub/app
SAFETYHUB_POSTGRES_DATA_DIR=/home/safetyhub/postgres
REPORTS_DIR=/app/data/reports
```

安装脚本会自动创建 `app`、`app/reports` 和 `postgres` 目录，PostgreSQL 数据会持久化在 `/home/safetyhub/postgres`，日报/周报/月报的 PDF、XLSX、CSV 文件会持久化在 `/home/safetyhub/app/reports`。

默认对外端口配置：

```env
SAFETYHUB_HTTP_PORT=80
```

这里的 `80` 是宿主机对外 HTTP 端口，容器内 Nginx 也监听 80，Compose 会做 `80:80` 端口映射。

如果内网统一入口已经有 HTTPS 网关、负载均衡或反向代理负责证书终止，SafetyHub 这边不需要额外放证书，只保持 `SAFETYHUB_HTTP_PORT=80`，并要求上游反代转发到 `http://<内网节点>:80` 且注入 `X-Forwarded-Proto: https`。

当前开发/内测服务器如果需要跑在 `4000`，只修改仓库根目录 `.env`，不要修改 `/var/www/NF-SafetyHub/内网离线部署包/intranet.env`。后者是打包输入，会直接进入内网交付包。

如果要求 SafetyHub 容器本身直接提供 HTTPS 证书终止，则还需要额外调整 `nginx/nginx.conf`、挂载证书目录，并把 Compose 改成容器内监听 443；当前离线包默认不是这种模式。

离线镜像模式默认已经启用：

```env
SAFETYHUB_IMAGE=nf-safetyhub-safetyhub:latest
SAFETYHUB_SKIP_BUILD=true
```

## 四、安装脚本会做什么

`install.sh` 会自动执行：

- `docker load -i safetyhub-images.tar`；
- 检查包内 `NF-SafetyHub/.env` 是否存在；
- 调用 `NF-SafetyHub/交付运行手册/deploy_intranet_docker.sh`；
- 启动 PostgreSQL；
- 重建业务表前，先用 `pg_dump` 把现有 `api_keys` 表备份到持久化数据盘 `SAFETYHUB_DATA_DIR/apikey_backups/api_keys_YYYYMMDD_HHMMSS.sql`（附 `.sha256`）；若数据库中尚无 `api_keys` 表（首次部署）则跳过备份，若备份失败则中止部署、不执行任何重建；
- 保留已有 `api_keys` 表，删除并重建当前系统需要的其他业务表；
- 创建报表目录 `${SAFETYHUB_DATA_DIR}/reports`，容器内对应 `/app/data/reports`，并确保容器用户 `1000` 可写；
- 启动 SafetyHub 和 Nginx。

本次内网升级不使用 `SAFETYHUB_IMPORT_APIKEYS`，也不从 JSON/SQL 重新导入 API Key。执行前请确认内网服务器现有 PostgreSQL 数据目录配置正确，尤其是：

```env
SAFETYHUB_POSTGRES_DATA_DIR=/home/safetyhub/postgres
POSTGRES_DB=safetyhub
POSTGRES_USER=safetyhub
POSTGRES_PASSWORD=现有内网数据库密码
DB_URL=postgresql+asyncpg://safetyhub:现有内网数据库密码@postgres:5432/safetyhub
```

如果这些值指向了新的空目录或不同数据库，部署脚本会看不到原来的 `api_keys` 表。

## 五、升级替换正在运行的实例（低中断）

本节面向**服务器上已经有一个正在运行的 SafetyHub `docker compose`**，要用新版离线包替换它，并尽量缩短中断、避免误删 `api_keys`。

### 为什么不直接 `bash install.sh`

`install.sh` 内部会调用部署脚本，按设计**删除并重建除 `api_keys` / `runtime_settings` 外的其他业务表**。在已有实例上直接跑，会有两个问题：

- 部署脚本不会先停掉仍在运行的旧 `safetyhub-app`，重建表时旧实例可能仍在写库，造成脏写竞争；
- 部署脚本最后是 `docker compose up -d safetyhub nginx`，`nginx` 容器配置没变不会被重建，进程内仍缓存旧 `safetyhub-app` 的 IP，新 app 重建后 IP 变化会持续 502。

所以替换运行中的实例时，要在部署脚本前后各补一步：**前置 `stop safetyhub`**、**后置 `restart nginx`**。

### 操作步骤

先按「二、上传到内网服务器」「三」的解包步骤把**新包**解到 `/opt`，但**不要**执行 `bash install.sh`，改用下面的序列：

```bash
cd /opt/safetyhub_intranet_bundle_<新时间戳>/NF-SafetyHub

# 0) 导入新镜像（postgres / nginx 不受影响）
docker load -i ../safetyhub-images.tar

# 1) 兜底：在部署脚本之外，先独立把 api_keys 备份一份到 /opt（双保险，防误删）
source .env
docker compose exec -T postgres env PGPASSWORD="$POSTGRES_PASSWORD" \
  pg_dump -h 127.0.0.1 -U "$POSTGRES_USER" -d "$POSTGRES_DB" \
  --data-only --column-inserts --table=public.api_keys \
  > /opt/api_keys_manual_$(date +%Y%m%d_%H%M%S).sql
# 确认备份文件非空再继续
ls -l /opt/api_keys_manual_*.sql

# 2) 先停旧 app，避免清表时脏写（postgres / nginx 不停）
docker compose stop safetyhub

# 3) 跑部署脚本：内部会再自动备份一次 api_keys → 清表重建 → 拉起新 safetyhub+nginx
bash 交付运行手册/deploy_intranet_docker.sh

# 4) 刷新 nginx 对 safetyhub 的 DNS 解析（app 容器重建后 IP 变了，不刷会 502）
docker compose restart nginx
```

要点：

- **第 1 步是关键的防误删保险**，独立于部署脚本：即使脚本任何环节出问题，`/opt` 里这份 `api_keys` 仍在。看到文件非空再往下走。
- 部署脚本第 3 步内部还会**再自动备份一次** `api_keys` 到持久化数据盘 `SAFETYHUB_DATA_DIR/apikey_backups/`（见第四节），且备份失败会中止、不执行任何重建。两份备份位置不同，互为冗余。
- **第 4 步 `restart nginx` 必须做**，否则替换后会持续 502。
- 中断窗口 = 第 2 步停 app 到第 3 步新 app 健康那几十秒；`postgres` 全程不停，`/home/safetyhub/postgres` 数据卷复用，`api_keys` / `runtime_settings` 保留。

### 禁止事项

- **不要** `docker compose down -v`，也**不要**删 `/home/safetyhub/postgres` —— 那会连同 PostgreSQL 数据卷一起删除，`api_keys` 也会丢。
- 不要在旧包目录和新包目录之间反复 `up` / `down`，容易把 compose 状态搞乱；选定**新包目录**作为唯一操作目录。

### 恢复 api_keys（万一需要回灌）

```bash
source .env
docker compose exec -T postgres env PGPASSWORD="$POSTGRES_PASSWORD" \
  psql -h 127.0.0.1 -U "$POSTGRES_USER" -d "$POSTGRES_DB" \
  < /opt/api_keys_manual_<时间戳>.sql
```

## 六、验证

默认对外端口是 443：

```bash
curl -i http://127.0.0.1/health/ready
curl -I http://127.0.0.1/admin/
```

浏览器通过 VPN 访问：

```text
http://内网服务器IP/admin/
```

如果内网网关或负载均衡已经配置 HTTPS 证书，用户侧访问地址通常是：

```text
https://内网域名/admin/
```

如果 `.env` 中改成其他 `SAFETYHUB_HTTP_PORT`，验证端口也要同步改。

## 七、常用运维命令

进入应用目录：

```bash
cd /opt/safetyhub_intranet_bundle_YYYYMMDD_HHMMSS/NF-SafetyHub
```

查看状态：

```bash
docker compose ps
```

查看日志：

```bash
docker compose logs -f safetyhub
```

重启：

```bash
docker compose restart safetyhub nginx
```

停止：

```bash
docker compose down
```
