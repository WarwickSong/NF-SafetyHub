**目录职责说明**
- `内网离线部署包/`：应用层离线包产出区，负责生成 `safetyhub_intranet_bundle_*.tar.gz`，包内包含 SafetyHub 代码、Compose 配置、`.env`、业务镜像和依赖镜像。这个包默认假设目标服务器已经有 Docker / Docker Compose。
- `docker离线部署/`：整机离线部署工具区，负责准备 Docker Engine、docker compose v2、systemd 单元和最终 `docker-offline-deploy_*.tar.gz`。它会把 `内网离线部署包/` 生成的 `safetyhub_intranet_bundle_*.tar.gz` 作为输入，适用于目标服务器没有 Docker / Docker Compose 的场景。
- `docker离线部署/部署文件/`：最终对外交付的 `docker-offline-deploy_*.tar.gz` + `.sha256` 默认产出位置（git 仅保留 `.gitignore` 与 `*.sha256`）。
- 推荐 Git 只管理脚本、配置、说明和 `.sha256`，不要提交 `*.tar`、`*.tar.gz`、Docker 二进制、镜像包和最终交付包。

**两段式交付链路（路径全部相对于仓库根，可整体迁移）**

```
内网离线部署包/intranet.env  ─┐
                              ├─► 交付运行手册/build_intranet_offline_bundle.sh
                              │       └─► 内网离线部署包/safetyhub_intranet_bundle_<ts>.tar.gz(.sha256)
                              │
                              └─► docker离线部署/scripts/prepare-offline-package.sh
                                      ├─ 下载 docker / compose 静态二进制 → docker离线部署/docker/
                                      ├─ 复制最新 safetyhub_intranet_bundle_*.tar.gz → docker离线部署/app-bundle/
                                      └─► docker离线部署/部署文件/docker-offline-deploy_<ts>.tar.gz(.sha256)
```

**你可以在仓库根目录下生成一个完整“内网离线部署包”，上传这一个包到内网服务器，然后运行包里的 `install.sh`。**

**现在的最简流程**
- 在当前服务器生成应用层 bundle（在仓库根执行，路径不再硬编码 `/var/www`）：
  
  ```bash
  cd <仓库根>
  bash 交付运行手册/build_intranet_offline_bundle.sh
  ```

- 生成位置默认是：
  
  ```text
  <仓库根>/内网离线部署包/safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz
  <仓库根>/内网离线部署包/safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz.sha256
  ```

- 如果目标内网服务器已经预装 Docker / Compose，可以直接把上面两个文件复制到目标服务器 `/opt`（或任意目录）后执行：
  
  ```bash
  cd /opt
  sha256sum -c safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz.sha256
  tar -xzf safetyhub_intranet_bundle_YYYYMMDD_HHMMSS.tar.gz
  cd safetyhub_intranet_bundle_YYYYMMDD_HHMMSS
  bash install.sh
  ```

- 第一次运行会生成 `NF-SafetyHub/.env` 并提示你补齐密码/认证版本；补齐后再执行：
  
  ```bash
  bash install.sh
  ```

**当目标服务器“连 Docker 都没有”时，再走整机离线包**

如果内网节点没有 Docker / Compose，需要把 Docker Engine 也一起带过去，使用 `docker离线部署/` 这一层：

```bash
cd <仓库根>/docker离线部署
bash scripts/prepare-offline-package.sh
```

脚本默认产物位置（路径都基于脚本所在目录推导，不依赖 `/var/www`）：

```text
<仓库根>/docker离线部署/部署文件/docker-offline-deploy_YYYYMMDD_HHMMSS.tar.gz
<仓库根>/docker离线部署/部署文件/docker-offline-deploy_YYYYMMDD_HHMMSS.tar.gz.sha256
```

把这两个文件拷到内网节点后：

```bash
sha256sum -c docker-offline-deploy_YYYYMMDD_HHMMSS.tar.gz.sha256
tar -xzf docker-offline-deploy_YYYYMMDD_HHMMSS.tar.gz
cd docker离线部署
sudo bash scripts/install.sh
```

可选环境变量（用于不同 cwd / 不同仓库位置）：

| 变量 | 用途 | 默认值 |
| --- | --- | --- |
| `PKG_OUTPUT_DIR` | 最终交付产物输出目录 | `docker离线部署/部署文件/` |
| `NF_PROJECT_ROOT` | NF-SafetyHub 仓库根（独立拷出 `docker离线部署/` 时使用） | `docker离线部署/` 的上级目录 |
| `DOCKER_VERSION` / `COMPOSE_VERSION` | docker / compose 版本 | `27.3.1` / `v2.29.7` |
| `ARCH` / `COMPOSE_ARCH` | 二进制架构 | `x86_64` / `x86_64` |

**公网网关 HTTPS 终止（生产强制要求）**
- 管理后台启用了 `Secure` Cookie，浏览器必须通过 `https://llm-safetyhub.nanfu.com` 访问，否则在 `/admin/api_keys.html` 等页面执行 PATCH/POST 时会出现 401 → 前端卡在“保存中...”。
- HTTPS 证书部署在公网网关/负载均衡上，业务服务器 Docker 容器内的 nginx 只监听 HTTP 80。
- 运维网关后端必须配置为 `http://<内网节点>:80`，不要配置成 `https://<内网节点>`。
- 网关转发到业务服务器时必须保留 `Host`，并注入 `X-Forwarded-Proto: https`。
- 健康检查路径使用 `GET http://<内网节点>/health/ready`。

**离线包里包含**
- `safetyhub-images.tar`：包含三个镜像：
  - `nf-safetyhub-safetyhub:latest`
  - `postgres:16-alpine`
  - `nginx:1.27-alpine`
- `NF-SafetyHub/`：完整运行代码、Compose、Nginx 配置和交付脚本。
- `install.sh`：内网服务器一键安装入口。
- 默认不再包含或导入 `safetyhub_apikeys_*.sql`，避免升级部署覆盖内网已有 PostgreSQL 数据。

**脚本会自动做什么**
- 自动执行 `docker load -i safetyhub-images.tar`。
- 检查包内 `NF-SafetyHub/.env` 是否存在。
- 自动启动 PostgreSQL、SafetyHub、Nginx。
- 自动执行数据库整理：保留内网服务器已有 `api_keys` 表，删除并重建当前系统需要的其他业务表。
- 最后输出 `docker compose ps` 和健康检查结果。
