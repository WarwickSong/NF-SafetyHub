- [ ] Request ID、user_id、api_key_id、model、capability 能贯穿归档和审计

#### 性能检查

- [ ] 中继转发额外延迟 < 50ms
- [ ] 安全检测耗时 < 10ms/请求
- [ ] 50 并发 SSE 连接稳定
- [ ] SQLite 查询在 10 万条数据时 < 200ms

#### 安全检查

- [ ] 绕过路径测试全部拦截（URL 编码、分段、Unicode 混淆等至少 3 种尝试）
- [ ] 日志中不包含 prompt 原文

- [ ] 管理后台需要认证
- [ ] 管理员查看详情、导出、规则操作写入管理员操作审计
- [ ] 数据库文件权限正确（仅 app 用户可读写）
- [ ] 文件上传入口默认关闭或限制大小/类型，不可解析文件按高风险处理
- [ ] 没有对外开放非必要端口

#### 运维检查

- [ ] TLS 证书自动续签配置正确

## 七、关键路径分析

```
阶段 0: 基础设施 + 运行基线 (1.5-2天)
    │
阶段 1: 核心安全链路 + 治理预留 (6-7天)
    ├── 检测引擎 (2天) ──────── 关键路径
    ├── 中继引擎 (1.5天) ────── 关键路径
    └── 消息归档 (1天)
    │
阶段 2: 管理能力 (5天)
    ├── 审计追溯 (1.5天)
    ├── 管理后台 (2天) ──────── 关键路径
    └── 告警通知 (1.5天)
    │
阶段 3: 打磨上线 (5天)
    ├── 测试 (2天) ──────────── 关键路径
    ├── DLP 规则完善 (1天)
    └── 部署 (1.5天) ────────── 关键路径
```

**关键路径总长**：阶段 0(1.5-2 天) → 检测引擎(2 天) → 中继引擎与治理预留(2.5 天) → 管理后台(2.5 天) → 测试(2.5 天) → 部署(2 天) = **13-14 天**

> 关键路径上的任务必须优先保障，任何延迟都会直接影响上线时间。

## 八、风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| SSE 长连接压垮服务器 | 低 | 高 | Nginx 限流 + Uvicorn worker 数量控制 + 阶段 3 压力测试 |
| 规则误杀正常对话 | 中 | 中 | 阶段 3 规则回归测试 + 上线后先以 `warn` 模式观察一周再切换 `block` |
| 规则被绕过 | 高 | 高 | 阶段 3 安全加固审查专门检查绕过路径（URL 编码、分段绕过、Unicode 混淆） |
| 数据库性能瓶颈 | 低 | 中 | SQLite 百万条以内够用，后续有明确的 PostgreSQL 迁移路径 |
| Python 内存泄漏 | 低 | 中 | Uvicorn `--workers 4` 且配置 `max-requests` 自动重启 worker |
| 正则 ReDoS 攻击 | 中 | 高 | 设置匹配超时（100ms）+ 对恶意构造文本降级放行 |
| 上游中转站不可用 | 中 | 中 | SafetyHub 不存储中转站状态，直接透传错误信息给客户端 |
| APIKey 明文泄露 | 中 | 高 | 只存哈希和前后缀，日志、告警、后台列表统一脱敏 |
| 权限体系后补导致重构 | 高 | 中 | v1.0 预留 api_key_id、model、capability、upstream_router，不启用复杂策略 |
| function call/MCP 伪装误触发工具 | 中 | 高 | 拦截时默认返回普通 assistant 文本，不生成 tool call，不模拟 MCP 工具结果 |
| 临时审批滥用 | 中 | 高 | 只允许可审批规则，令牌一次性、短有效期、绑定请求哈希和审批审计 |
| 文件上传绕过检测 | 高 | 高 | 文件入口默认关闭或白名单启用，不可解析文件默认拒绝或审批 |
