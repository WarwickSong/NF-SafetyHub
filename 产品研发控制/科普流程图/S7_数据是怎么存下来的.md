# S7. 数据是怎么存下来的（一句话版）

> 为什么"存档"不会让员工感觉变慢——一张图说清。

```mermaid
flowchart LR
    User(["员工请求"])
    User --> Hub["前哨站处理<br/>(扫描 / 转发 / 收回答)"]
    Hub --> Quick["立刻把回答<br/>原样发回给员工"]
    Quick --> Done(["员工拿到回答 ⏱"])

    Hub -. "顺手丢进一个篮子" .-> Basket[("内存里的<br/>归档篮子")]
    Basket -. "后台小工每秒打包一次" .-> Worker["后台小工<br/>批量入库"]
    Worker -. "一次写一批" .-> DB[("数据库<br/>PostgreSQL")]

    classDef fast fill:#dff0d8,stroke:#27ae60,color:#000
    classDef slow fill:#e3f2fd,stroke:#1976d2,color:#000
    class Hub,Quick,Done fast
    class Basket,Worker,DB slow
```

## 一句话

> **员工的回答先走，数据存档由后台慢慢做——员工感觉不到任何延迟。**

## 三个保护设计

| 场景 | 设计 |
|------|------|
| 高峰期请求暴涨 | 篮子有容量上限，满了就丢一部分（保住用户体验），并在后台看板可见"丢弃数" |
| 后台写库失败 | 不影响员工请求，只是这一批数据落不了，会被记录 |
| 系统重启 | 关闭前先等篮子排空再退出，避免丢数据 |

## 为什么要这么设计

如果让"存档"和"回答员工"串在一起做，数据库一卡顿，员工就觉得 AI 卡了。
现在拆成两条路：**主路只管回答，存档走副路。**
